# EGCE v0.2 方案：研发全流程上下文引擎

## 1. 定位

EGCE 是一个**全局安装的命令行工具库**，为 AI 编码助手（Claude Code / Cursor / Codex）提供：

- 代码仓库的结构化上下文
- 基于代码的需求分析能力
- 结构化的 AI 需求规格（spec）
- 研发全流程的上下文管理

使用者不需要懂代码，不需要懂 EGCE 的内部实现。所有操作通过 AI 对话完成。

## 2. 用户体验

### 2.1 首次使用

```
用户：pip install egce（一次性，全局安装）
用户：打开 Claude Code / Cursor / Codex
用户：我要开发 xxx 项目，前端仓库是 git@xxx/frontend.git，后端仓库是 git@xxx/backend.git
  AI：请问你想把工作区创建在哪个位置？
用户：~/workspace/my-project
  AI：好的，正在创建工作区...
      → mkdir ~/workspace/my-project
      → git clone frontend
      → git clone backend
      → egce init（自动扫描，生成 .egce/ 结构）
  AI：工作区创建完成。我已经扫描了前后端代码，接下来我帮你生成项目上下文。
      → AI 读取扫描结果，通过对话生成 context 文件
  AI：项目上下文已生成，你可以开始工作了。
```

### 2.2 日常使用（不区分角色）

```
PM：     "我想加一个批量导出功能，帮我分析一下怎么做"
开发：   "按这个需求规格开始开发后端第一个任务"
测试：   "这次改动影响了哪些功能？帮我列一下测试点"
新人：   "帮我介绍一下这个项目的架构"
```

所有人用同一个工具，说自然语言，AI + EGCE 在后面干活。

## 3. 目录结构

### 3.1 工作区（本地，不提交到 Git）

```
~/workspace/my-project/          ← 工作区根目录
  .egce/                         ← 工作区级配置（本地）
    workspace.yaml               ← 工作区配置：包含哪些项目
  backend/                       ← 后端仓库（git clone）
    .egce/                       ← 后端项目上下文（随代码提交）
    src/
    ...
  frontend/                      ← 前端仓库（git clone）
    .egce/                       ← 前端项目上下文（随代码提交）
    src/
    ...
```

### 3.2 workspace.yaml（工作区级，本地）

```yaml
# 工作区配置 — 告诉 EGCE 这个工作区包含哪些项目
workspace: my-project
projects:
  - name: backend
    path: ./backend
    language: python
  - name: frontend
    path: ./frontend
    language: typescript
```

### 3.3 各仓库内的 .egce/（随代码提交到 Git）

```
backend/.egce/
  config.yaml                    ← 项目配置
  context/                       ← 项目上下文（核心价值）
    architecture.md              ← 架构说明
    modules.md                   ← 模块职责和边界
    conventions.md               ← 开发约定
    api-contracts.md             ← 本项目暴露的 API 接口
    data-models.md               ← 数据模型说明
  specs/                         ← AI 需求规格
    feature-batch-export.yaml    ← 某个需求的规格
    ...
  CLAUDE.md                      ← 本项目的 AI 工作指令
```

### 3.4 config.yaml（项目级）

```yaml
project: backend
language: python
framework: fastapi

# 代码扫描配置
scan:
  include: ["src/*", "app/*"]
  exclude: ["tests/*", "docs/*", "migrations/*"]

# 测试配置
verify:
  test: "pytest --tb=short -q"
  lint: "ruff check ."

# 上下文加载优先级
context_priority:
  - context/architecture.md
  - context/api-contracts.md
  - context/modules.md
  - context/conventions.md
  - context/data-models.md
```

## 4. AI 需求规格（Spec）

### 4.1 为什么不用传统 PRD

传统 PRD 是给人看的，信息是模糊的：
- "用户可以批量导出数据" → 什么接口？什么字段？什么格式？

AI 需求规格是给 AI 执行的，信息是精确的：
- 接口路径、方法、请求体、响应体、字段类型全部定义
- 前端改哪个页面、加什么组件、调什么接口全部指定
- 测试用例明确列出

### 4.2 Spec 格式

```yaml
# backend/.egce/specs/feature-batch-export.yaml

id: feature-batch-export
title: 批量导出功能
description: 用户在列表页勾选多条记录，点击导出按钮下载 CSV 文件
status: draft            # draft → approved → in_progress → done
created_at: 2026-04-01
approved_by: null        # 人工审核后填写

# ── 后端任务 ──
backend:
  tasks:
    - id: be-1
      title: 新增批量导出接口
      status: pending
      api:
        method: POST
        path: /api/v1/exports/batch
        request:
          body:
            ids:
              type: list[int]
              description: 要导出的记录 ID 列表
            format:
              type: string
              enum: ["csv", "xlsx"]
              default: "csv"
        response:
          200:
            task_id:
              type: string
              description: 异步任务 ID
          400:
            error:
              type: string
      affected_files:
        - src/api/exports.py         # 新增
        - src/services/export_service.py  # 新增
      depends_on:
        - src/services/query_service.py   # 已有

    - id: be-2
      title: 新增导出状态查询接口
      status: pending
      api:
        method: GET
        path: /api/v1/exports/{task_id}/status
        response:
          200:
            status:
              type: string
              enum: ["pending", "processing", "done", "failed"]
            download_url:
              type: string
              nullable: true
      affected_files:
        - src/api/exports.py         # 修改

  testing:
    - 批量导出接口：传入 10 条 ID，返回 task_id
    - 状态查询接口：返回正确状态
    - 空 ID 列表返回 400
    - 超过 1000 条返回 400

# ── 前端任务 ──
frontend:
  tasks:
    - id: fe-1
      title: 列表页增加批量导出按钮和交互
      status: pending
      pages:
        - path: /records
          changes:
            - 列表顶部增加"批量导出"按钮
            - 勾选 >= 1 条记录时按钮激活
            - 点击后调用 POST /api/v1/exports/batch
            - 显示进度条，轮询 GET /api/v1/exports/{task_id}/status
            - status=done 后自动触发下载
      api_calls:
        - POST /api/v1/exports/batch
        - GET /api/v1/exports/{task_id}/status
      affected_files:
        - src/pages/records/RecordList.tsx    # 修改
        - src/services/exportApi.ts           # 新增
        - src/components/ExportButton.tsx     # 新增

  testing:
    - 未勾选时按钮禁用
    - 勾选后按钮可点击
    - 点击后显示进度
    - 完整导出流程走通
```

### 4.3 Spec 的生命周期

```
PM 描述需求（自然语言）
  ↓
AI + EGCE 分析现有代码 → 输出 spec 草稿（status: draft）
  ↓
人审核 → 修改 → 标记 approved
  ↓
AI 按 spec 逐任务开发（status: in_progress）
  ↓
每个任务完成后 → egce verify → 标记任务 done
  ↓
全部任务完成 → spec status: done
  ↓
更新 .egce/context/（如果有新模块、新接口）
  ↓
提交代码 + .egce/ 一起入库
```

## 5. EGCE 新增命令

### 5.1 egce init

```bash
egce init [path]
```

扫描当前目录（或指定目录），生成 `.egce/` 结构：

- 检测目录下的 Git 仓库
- 检测语言和框架
- 生成 `config.yaml`
- 创建 `context/` 目录（空文件，带模板注释）
- 创建 `specs/` 目录
- 生成 `CLAUDE.md`
- 如果检测到多个仓库，额外生成 `workspace.yaml`

**不调 LLM。** context 文件的内容由 AI 在对话中生成。

### 5.2 egce sync

```bash
egce sync [path]
```

重新扫描代码，更新自动生成的内容：
- 重新生成 repo-map
- 检查 config.yaml 中的路径是否仍然有效
- 检查 specs 中引用的文件是否存在

### 5.3 egce spec

```bash
egce spec list                    # 列出所有 spec 及状态
egce spec show <spec-id>          # 显示某个 spec 详情
egce spec status <spec-id> <status>  # 更新状态
```

管理 `.egce/specs/` 目录下的需求规格。

### 5.4 egce context

```bash
egce context list                 # 列出当前项目的上下文文件
egce context show <name>          # 显示某个上下文文件
```

查看 `.egce/context/` 目录内容。

### 5.5 已有命令保持不变

```bash
egce scan       # 扫描仓库结构
egce search     # 搜索相关代码
egce pipeline   # 全流程：检索 → 压缩 → 打包
egce verify     # 运行测试/lint
egce pack       # 手动组装 prompt
```

## 6. EGCE 内部改动

### 6.1 上下文加载

ContextPacker 在打包 prompt 时，自动加载 `.egce/context/` 里的文件作为高优先级 slot：

```
packer 槽位：
  system        → CLAUDE.md
  project_ctx   → .egce/context/*.md（按 config 中的优先级排序）
  task          → 用户任务描述
  spec          → 当前执行的 spec（如果有）
  repo_map      → 代码结构
  evidence      → 检索到的代码片段
  verifier_feedback → 测试结果
  output_contract → 输出格式要求
```

新增 `project_ctx` 槽位，优先级设为 HIGH，确保项目上下文不会被裁剪掉。

### 6.2 多项目索引

Retriever 支持同时索引多个项目：

```python
retriever = Retriever.from_workspace("~/workspace/my-project")
# 自动读取 workspace.yaml，索引所有子项目
# 搜索结果带项目标识：backend:src/api/exports.py
```

### 6.3 Spec 加载

search 和 pipeline 命令在有活跃 spec 时，自动将 spec 内容注入上下文：

```python
# 当有 status=in_progress 的 spec 时
packer.set_slot("spec", spec_content, priority=Priority.HIGH)
```

## 7. CLAUDE.md 模板

```markdown
# 项目工作指南

本项目使用 EGCE 上下文引擎。请严格按以下流程工作。

## 初始化工作区

当用户提供仓库地址要求创建工作区时：
1. 询问用户工作区创建位置
2. 创建目录并 git clone 各仓库
3. 执行 `egce init`
4. 读取扫描结果，为每个项目生成 context 文件
5. 让用户确认 context 内容

## 需求分析阶段

当用户描述一个需求时：
1. 用 `egce search` 在各项目中搜索相关代码
2. 读取 .egce/context/ 了解现有架构和约定
3. 基于代码和上下文分析需求可行性
4. 输出结构化 spec（yaml 格式）到 .egce/specs/
5. 请用户审核 spec

重要：spec 中必须包含完整的接口定义、字段定义、受影响文件列表。
不要输出模糊描述，要输出 AI 可执行的精确规格。

## 开发阶段

当用户批准 spec 后：
1. 按 spec 中的任务列表逐个执行
2. 每个任务开始前，用 `egce pipeline` 获取相关上下文
3. 严格按 spec 中的接口定义编码，不要自行修改接口
4. 每个任务完成后执行 `egce verify`
5. 验证通过后更新 spec 中的任务状态

## 上下文维护

当一个 spec 完成后：
1. 如果引入了新模块，更新 .egce/context/modules.md
2. 如果新增了接口，更新 .egce/context/api-contracts.md
3. 如果有新的开发约定，更新 .egce/context/conventions.md
4. 提交 .egce/ 目录变更到 Git
```

### Phase 3：完善

- [ ] `egce context` 子命令

## 9. 不做的事

- 不做 Web UI — AI 工具就是 UI
- 不调 LLM — EGCE 是纯本地工具，LLM 调用交给 AI 工具
- 不做 Git 操作 — clone / commit / push 交给 AI 工具
- 不做权限管理 — 用 Git 仓库权限
- 不做项目管理 — 用 spec status 做最简单的状态跟踪，复杂的用 Jira/Linear
